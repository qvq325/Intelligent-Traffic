# 禁停场景全局摄像头绑定设计

日期：2026-07-14
状态：已批准

## 背景

当前 `scene_archive` 同时保存禁停场景和道路异常场景，两类场景都被要求记录
`topology_id`、`topology_revision` 和 `review_status`。禁停场景实际只依赖视频画面中的
检测区域和固定摄像头，拓扑修订变化不应使它失效。现有耦合会导致旧禁停场景在切换或
编辑道路拓扑后返回 `SCENE_TOPOLOGY_MISMATCH`，并且拓扑更新会停止正在运行的禁停监控。

## 目标

- 禁停场景只绑定固定 12 路全局摄像头中的一个。
- 禁停场景不再绑定道路拓扑或拓扑修订。
- 现有禁停场景自动迁移，保留名称、参考图、检测区域和摄像头。
- 道路拓扑更新、切换和删除不再使禁停场景失效或停止禁停监控。
- 道路异常场景继续保留现有拓扑绑定和复核规则。
- 新旧配置包均可导入，并在导入时统一为新的禁停场景语义。

## 非目标

- 不改变固定摄像头目录及其 12 路约束。
- 不改变流关联方案结构；场景仍通过当前激活流方案解析摄像头视频流。
- 不取消道路异常场景与道路拓扑的关系。
- 不移除禁停页面中的拓扑可视化。该视图可以使用当前拓扑展示全局摄像头位置，但不会
  将当前拓扑写入禁停场景。

## 方案选择

采用共享场景表的条件约束方案。`scene_archive` 继续统一保存两类场景，但禁停场景的
`topology_id` 和 `topology_revision` 为 `NULL`，道路异常场景的两个字段仍为必填。

未采用以下方案：

- 拆分两张场景表：领域边界更明确，但会显著扩大 API、激活状态和配置包的改造范围。
- 仅跳过启动校验：数据库外键、拓扑删除保护、复核状态和导入导出仍会保留隐性耦合，
  无法永久解决问题。

## 数据模型与迁移

配置数据库升级到 schema v3，并重建 `scene_archive`：

- `camera_id` 保持非空，并继续外键关联固定 `camera` 表。
- `topology_id` 和 `topology_revision` 改为可空。
- 增加按场景类型约束：
  - `no_parking`：两个拓扑字段必须为 `NULL`，`review_status` 为 `ready`。
  - `road_abnormal`：两个拓扑字段必须非空，修订号至少为 1。
- 保留拓扑、摄像头和参考图外键；SQLite 允许空拓扑外键。
- 迁移现有 `no_parking` 行时将两个拓扑字段清空，并将 `review_status` 设为 `ready`。
- 迁移现有 `road_abnormal` 行时原样保留拓扑字段和复核状态。
- 保留激活状态中的 `no_parking_scene_id`，因此迁移不会丢失当前选中的禁停场景。

迁移在单个数据库事务内完成。迁移后运行外键检查和完整性检查；任一步失败时回滚，旧库
保持不变。

## 保存与读取

`ApplicationState.persist_scene` 按场景类型构造持久化数据：

- 禁停场景写入 `topology_id = NULL`、`topology_revision = NULL`。
- 道路异常场景继续写入当前激活拓扑及其修订。

`ConfigurationService.upsert_scene_archive` 在写入前校验领域规则，并依靠摄像头外键保证
`camera_id` 只能来自固定全局目录。读取接口继续返回统一场景结构；禁停场景的拓扑字段
明确返回 `null`。

## 激活规则

激活禁停场景时依次检查：

1. 场景存在且类型正确。
2. `camera_id` 存在于固定摄像头目录。
3. 当前激活流方案提供该摄像头的可用流。
4. 场景流通过首帧预检。
5. 禁停运行通道成功启动。

禁停场景不再检查 `review_status`、拓扑 ID 或拓扑修订。道路异常场景继续执行原有复核和
拓扑兼容检查。

## 拓扑生命周期

- 更新拓扑时，只将引用旧修订的道路异常场景标记为 `needs_review`。
- 更新当前拓扑时，只清除失效的 `road_abnormal_scene_id`；保留
  `no_parking_scene_id`。
- 切换拓扑时，只停用不兼容的道路异常场景；正在运行的禁停监控保持运行。
- 保存道路态势修改时，不再停止禁停监控及其视频通道。
- 删除非激活拓扑时，只检查道路异常场景引用，禁停场景不再阻止删除。
- 拓扑切换失败并回滚时，禁停监控不参与拓扑回滚过程。

## 配置包兼容

场景文档继续保留 `topology_id` 和 `topology_revision` 键，避免无必要的文件格式分叉：

- 新导出的禁停场景将两个值写为 `null`。
- 新导出的道路异常场景继续写入实际拓扑值。
- 导入旧包时，忽略禁停场景中的旧拓扑值并规范化为 `null`。
- 导入道路异常场景时继续验证拓扑存在、修订兼容和复核状态。
- 配置包预检中的激活场景兼容性校验按场景类型执行；禁停场景只校验摄像头和流绑定。

## 界面调整

禁停编辑器已经只提交 `camera_id`，无需改变保存表单。系统管理的场景档案列表调整为：

- 禁停场景的关系列显示“全局摄像头”，复核状态显示“无需拓扑复核”。
- 禁停场景激活按钮不再受 `needs_review` 限制。
- 禁停场景激活确认框只展示摄像头，不展示拓扑和修订。
- 道路异常场景继续展示拓扑、修订和复核状态。

通用 API 错误解析同时读取 `error.message`，确保后续配置错误不会再次退化成泛化的
“请求失败 (422)”。

## 错误处理

- 摄像头不在固定目录：返回结构化 `SCENE_CAMERA_INVALID`，HTTP 422。
- 当前流方案缺少摄像头映射：返回 `SCENE_STREAM_MISSING`，HTTP 422。
- 场景流预检失败：返回 `STREAM_PROBE_FAILED`，HTTP 422，并附带探测详情。
- 道路异常场景拓扑不兼容：继续返回 `SCENE_TOPOLOGY_MISMATCH`。
- 数据库迁移失败：启动失败并保留旧事务状态，不部分应用 schema v3。

## 测试策略

- schema v2 到 v3 迁移：禁停字段清空、内容保留、道路异常字段不变、外键检查通过。
- 新库约束：禁停场景只能引用固定摄像头且不能写入拓扑；道路异常必须写入拓扑。
- 禁停启动：旧迁移场景和新场景都能在任意当前拓扑下启动。
- 拓扑更新与切换：禁停场景保持激活，道路异常场景按原规则失效。
- 拓扑删除：禁停场景不阻止删除，道路异常引用仍会阻止删除。
- 配置包：新格式往返一致，旧包禁停场景自动规范化，道路异常非法引用仍被拒绝。
- 前端：禁停场景显示全局摄像头语义，真实结构化错误消息能够展示。
- 回归：现有禁停检测、道路异常检测、配置仓库、配置 API 和前端测试全部通过。

## 验收标准

- 四个现有禁停场景迁移后均保留原配置并可以启动。
- 修改或切换道路拓扑后，禁停场景无需重新保存或复核。
- 禁停场景始终且仅绑定一个固定全局摄像头。
- 道路异常场景的拓扑安全约束没有放宽。
- 服务重启、配置导入导出和系统管理界面均体现相同语义。
