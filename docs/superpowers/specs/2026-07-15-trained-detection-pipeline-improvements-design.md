# 训练后检测流水线效果优化设计

## 目标

在不重复接入模型、不改变现有 API 契约的前提下，将 `improve/` 中已经验证的车辆、车牌和七层 MOG 策略正确应用到生产链路：

- 提高训练后车牌检测的召回率和跨帧稳定性。
- 降低道路异常场景将正常移动车辆识别为障碍物的概率。
- 消除道路异常场景的重复模型加载和非原始帧输入。
- 保持 legacy 流水线、配置导入导出、事件记录和其他场景兼容。

本次采用折中方案：允许适度增加 GPU 推理，但通过批量补检、单一推理所有者和按需复核控制显存与延迟。目标运行设备为当前项目使用的 NVIDIA GeForce RTX 4060 Laptop GPU（8 GiB）。

## 已确认现状

- `训练后模型/yolo26x.pt` 与 `训练后模型/license_plate_best.pt` 已由可信模型注册表接入，并可通过 `trained` 预设启用。
- 当前运行配置已为四个固定场景选择 `trained` 预设；实时场景车牌阈值为 `0.3`。
- `BoxPlateRecognizer` 当前只做一次全帧车牌检测，没有车辆裁剪补检、检测框几何过滤或跨帧投票。
- `backend/trained_mog.py` 已移植七层 MOG 策略，但生产事件层会在 MOG 确认后再次完整等待场景持续时间。
- 道路异常视频服务和 `RoadAbnormalMonitor` 都可能持有检测流水线；配置加载顺序还可能重新启用通用 `DetectionProcessor`，造成重复模型与标注帧进入 MOG。
- 现有道路异常事件中包含正常车辆画面对应的 `MOG` 小障碍物记录，说明需要当前帧正常车辆复核。

## 范围

### 包含

- 训练后车辆/车牌流水线的分级车牌检测和时序融合。
- 道路异常场景的单一推理所有权、原始帧处理、按需 YOLO 复核和持续时间衔接。
- 为上述行为增加单元、集成、并发和真实模型验证。
- 保持当前模型配置字段并复用 `yolo_threshold`、`lpr_threshold`、`frame_interval`、`inference_size` 和 `mog_*` 参数。

### 不包含

- 重新训练或修改任何权重文件。
- 修改 `improve/` 或 `训练后模型/` 中的算法副本。
- 新增模型上传、任意模型路径或数据库配置字段。
- 建立跨场景全局 GPU 调度器。
- 改变禁停规则、道路态势业务规则或 legacy 模型的识别语义。

## 架构

### 车辆与车牌

生产接口继续由 `DetectionProcessor.process(frame, camera_id)` 返回现有 `DetectionResult`。训练后 box 模式内部改为分级流水线：

```text
原始帧
  -> yolo26x 车辆检测与 ByteTrack
  -> license_plate_best 全帧检测
  -> 车牌/车辆空间关联
  -> 仅对未匹配车辆批量裁剪补检
  -> 坐标还原、几何过滤和去重
  -> 单次批量 OCR
  -> 摄像头 + track_id 时序融合
  -> 白名单匹配和 DetectionResult
```

`BoxPlateRecognizer` 保持 `recognize(frame)` 兼容入口，并增加供 `DetectionProcessor` 使用的车辆感知入口。legacy pose 模式仍调用原有识别器，不进入补检和投票逻辑。

### 道路异常

道路场景采用单一推理所有者：

```text
原始帧
  -> 每帧七层 MOG
  -> 定期 yolo26x 正常/异常类别检测
  -> MOG 告警前按需执行当前帧 YOLO 复核
  -> ROI 与正常车辆过滤
  -> YOLO/MOG 候选融合
  -> 现有事件、截图和告警流程
```

`road_abnormal_video` 进入外部推理所有者模式：它继续负责采集、发布、模型配置元数据和状态展示，但不创建通用 `DetectionProcessor`。`RoadAbnormalMonitor` 是道路场景唯一的 YOLO/MOG 所有者。

`VideoStreamService` 在调用帧分析器时传入未绘制检测框、信息面板或场景覆盖物的原始 BGR 帧。帧分析器返回的道路覆盖物再进入发布流程。

## 车辆与车牌数据流

### 全帧检测

- 每个推理帧只执行一次全帧车牌模型调用。
- 使用当前 `lpr_threshold` 作为全帧检测阈值。
- 检测框先裁剪到图像范围，再与车辆框进行中心点和覆盖关系关联。
- 同一车辆存在多个候选时，优先使用检测置信度更高且几何关系更合理的候选。

### 批量裁剪补检

- 只补检没有可靠全帧车牌候选的车辆。
- 车辆框按宽高各向外扩展 20%，并裁剪到图像边界。
- 一帧最多补检 16 辆车；超出时按车辆检测置信度和可见面积排序。
- 所有车辆裁剪组成一个批次，车牌模型每帧最多增加一次调用，不按车辆逐次调用。
- 补检阈值由 `lpr_threshold * 0.6` 派生，并限制在 `0.08-0.20`。
- 候选需要满足合理宽高比、最小像素尺寸、车牌面积不超过车辆面积的 35%，并且中心位于对应车辆区域内。
- 全帧与补检候选使用 IoU 和中心距离去重，保留综合置信度较高者。

### OCR 与时序融合

- 全帧与补检产生的最终候选合并后只进行一次批量 OCR。
- 内部候选保留车牌检测置信度、OCR 置信度、来源和关联的 `track_id`；公开的 `PlateRecognition` 与 `DetectionResult` 不增加字段。
- 投票键为 `(camera_id, track_id)`。无有效 `track_id` 的候选只返回当前帧结果，不进入跨帧缓存。
- 每个轨迹最多保留最近 5 次推理观察，文本按检测置信度与 OCR 置信度乘积进行精确字符串加权投票。
- 当前帧漏检时，稳定结果最多保留 5 次推理，置信度随缺失次数衰减；车牌框按车辆内相对坐标映射到当前车辆位置。
- 每个处理器最多保留 256 个轨迹缓存，按最近使用时间淘汰。
- 切换视频源、替换模型、重置跟踪或轨迹超过保持期时清理对应缓存。
- 白名单仍对最终稳定文本逐帧计算，避免缓存过期白名单状态。

## 道路异常数据流

### 参数归属

- 模型流水线配置控制设备、YOLO 阈值、推理间隔、输入尺寸和训练后 `mog_*` 参数。
- 道路场景配置继续控制 ROI、正常/异常类别、事件持续时间和丢失容忍时间。
- legacy MOG2 保留原有场景级背景参数和行为。
- 当模型流水线 `enabled=false` 时，视频仍可播放，但道路监控不执行 YOLO 或 MOG 推理。

### 常规检测与按需复核

- 训练后 YOLO 按模型流水线的 `frame_interval` 定期运行，结果更新正常车辆框和已知异常类别。
- 七层 MOG 每个输入帧运行，并继续使用近期正常车辆掩码抑制车辆残影。
- 当 MOG 产生可进入事件层的候选、且最近 YOLO 结果超过 2 帧时，对当前原始帧增加一次 YOLO 复核。
- 同一帧的所有 MOG 候选共享这次复核；不会按候选重复调用模型。
- 复核检测到正常车辆覆盖候选主体或包含候选中心时，过滤该 MOG 候选。
- 复核检测到配置中的异常类别时，使用现有 YOLO 候选结构进入事件层。
- 复核结果同时刷新近期正常物体缓存，避免下一帧立即重复推理。

### 持续时间衔接

- `TrainedMOGAlert` 增加仅供内部适配使用的已观察持续时间。
- MOG 候选进入事件层时，以该持续时间初始化候选的 `first_seen` 和 `duration_seconds`。
- 最终事件等待时间等于 MOG 过滤时长与场景事件持续时间的较大值，而不是两者相加。
- 公开事件字段、快照路径、候选状态和历史文件格式保持不变。

## 并发与资源边界

- 继续使用道路监控现有的推理锁和运行时 generation 检查，模型替换、场景切换和正在处理的帧不会交叉提交结果。
- 新模型和 MOG 引擎在状态锁外准备，成功后再原子替换；失败保留完整旧策略。
- 道路外部推理模式不会加载车牌模型或第二份车辆模型。
- 车牌补检始终批量执行，并限制每帧补检数和时序缓存大小。
- 告警活跃期间，按需 YOLO 复核最多每 2 帧一次；普通情况下仍遵守配置间隔。

## 错误处理

- 全帧车牌检测成功而补检失败时，保留全帧结果并记录降级状态。
- OCR 失败时不创建新的文本结果，但允许同轨迹已确认文本在保持期内继续显示。
- 道路定期 YOLO 失败时，MOG 继续运行。
- MOG 告警复核失败时采用 fail-open：保留 MOG 候选并报告复核降级，避免因辅助模型故障漏掉真实异常。
- MOG 或道路帧处理异常不终止视频采集与发布。
- 所有模型错误继续通过现有脱敏边界输出，不暴露可信权重的绝对路径或底层异常细节。

## 兼容性

- 不修改 SQLite schema、配置包格式、REST API 请求或响应模型。
- 不修改前端模型配置字段。
- legacy 车辆和 pose 车牌识别保持现有路径。
- `DetectionResult`、`PlateRecognition`、道路候选和事件 JSON 契约保持兼容。
- `VideoStreamService` 的外部推理所有者选项默认关闭，现有实时、道路态势和禁停视频服务不改变所有权。
- 训练后 box 识别的增强逻辑由共享处理器复用，但主要真实模型验收场景为实时车辆/车牌识别。

## 验证与验收

### 自动化测试

- 全帧车牌检测、只补检未匹配车辆、单次批量调用和裁剪坐标还原。
- 车牌尺寸、宽高比、面积、边界和重复候选过滤。
- 时序加权投票、短时漏检保持、摄像头/轨迹隔离、缓存上限和重置清理。
- legacy pose 路径不进入训练后补检逻辑。
- 道路场景不创建通用检测处理器，并且帧处理器收到原始像素。
- 常规 YOLO 间隔、复核频率限制、正常车辆抑制和未知障碍物保留。
- MOG 已观察时长与事件持续时间取最大值。
- 模型替换失败、并发场景切换和推理错误保持现有原子性。
- 运行完整 pytest、JavaScript 语法检查和 `git diff --check`。

### 真实模型验证

- 使用 `runtime/uploads/*_车牌识别.mp4` 对比变更前后的车辆关联车牌次数、同轨迹文字跳变次数和连续显示比例。
- 使用 `runtime/uploads/*_车辆追踪.mp4` 检查正常车辆不会形成道路 MOG 事件，并检查覆盖物稳定性。
- 固定抽取至少 300 个 1280x720 帧，记录端到端推理延迟的 P50/P95、处理 FPS 和 `torch.cuda.max_memory_allocated()`。
- 验证道路场景只持有一份 YOLO，且不加载车牌检测/OCR 模型。
- 在 RTX 4060 8 GiB 上持续运行完整样例视频，不出现 CUDA OOM、模型重复加载或视频线程退出。
- 启动本地服务，实际检查车辆/车牌叠加画面、道路异常候选、错误状态和模型状态展示。

真实视频没有完整人工标注，因此连续性、误报和性能数据作为前后对比报告，不伪装成精确率或召回率。行为正确性由确定性的单元与集成测试兜底。

## GitNexus 影响评估

- `DetectionProcessor`：MEDIUM，7 个直接依赖、19 个上游符号；主要波及视频服务、应用状态和检测相关测试，索引未识别到执行流。
- `BoxPlateRecognizer`：LOW，3 个直接依赖。
- `RoadAbnormalMonitor` 与 `RoadObjectDetector`：LOW，各 2 个直接依赖。
- `VideoStreamService._run`：LOW，无索引上游调用者，但属于共享线程循环，必须通过视频服务回归测试验证。
- `ApplicationState.apply_model_pipeline_settings` 与 `activate_scene_runtime`：LOW，影响启动配置加载和配置重载执行流。
- 没有 HIGH 或 CRITICAL 风险项。

实施前仍需对每个实际修改的方法执行精确的 upstream impact 查询；提交前使用 `detect_changes` 对照预期符号和执行流。
