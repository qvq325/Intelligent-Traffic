# 车辆视频分析模块开发者使用与系统集成指南

> 适用范围：当前目录中的 `detect_stream.py`、`mog_anomaly_engine.py`、
> `parking_zone.py`、`view_stream.py` 及两个本地模型权重。
>
> 文档定位：面向需要把该模块接入真实软件系统的开发人员。本文会明确区分
> “当前代码已经实现的能力”和“生产环境建议实现的能力”。

## 1. 模块概述

该模块是一个基于 OpenCV、Ultralytics YOLO、ByteTrack 和 MOG2 的视频分析原型。
它可以从摄像头、视频文件或网络视频流读取画面，执行以下处理：

| 能力 | 实现方式 | 默认启用 | 当前输出 |
|---|---|---:|---|
| 车辆及通用目标检测 | `yolo26x.pt` + Ultralytics YOLO | 是 | 画面标注、控制台统计 |
| 目标跟踪 | ByteTrack | 否 | 跟踪 ID、画面标注 |
| 车牌检测 | `license_plate_best.pt` | `both` 模式启用 | 车牌框 |
| 车牌字符识别 | HyperLPR3 | 否 | 画面文字 |
| YOLO 类别异常判断 | COCO 类别白名单 + 多帧投票 | 使用车辆模型时启用 | 红框、计数 |
| 背景异常检测 | OpenCV MOG2 + 规则过滤 | 否 | 告警框、控制台日志 |
| 禁停区检测 | 多边形区域 + ByteTrack 轨迹 | 否 | 违停框、控制台日志 |
| 截图 | OpenCV | GUI 模式可用 | `screenshots/*.jpg` |
| 视频写出 | OpenCV `VideoWriter` | 仅无 GUI 分支 | AVI/XVID 文件 |

### 1.1 当前定位

当前实现适合作为算法验证、单路视频演示和集成原型，不应直接视为完整的生产推理服务。
它尚未提供稳定的结构化结果接口、HTTP API、消息队列、配置文件、持久化、完善的重连机制、
健康检查或多路视频调度。

### 1.2 输入与输出

支持的输入：

- 本地摄像头，例如 `0`、`1`。
- 本地视频文件，例如 `D:\videos\road.mp4`。
- RTSP、RTMP、HTTP 或 HTTPS 视频地址。

当前输出：

- OpenCV 预览窗口及画面标注。
- 控制台状态和部分异常日志。
- GUI 模式下的手动截图。
- 无 GUI 分支下的视频文件。

当前不会输出：

- JSON 检测结果。
- HTTP/WebSocket/SSE 响应。
- Kafka、RabbitMQ、Redis Stream 等消息。
- 数据库记录。
- 自动保存的异常快照或车牌记录。

## 2. 目录与文件职责

```text
模型使用/
├── detect_stream.py          # 主入口：采集、推理、跟踪、规则、显示
├── mog_anomaly_engine.py     # MOG2 背景异常检测引擎
├── parking_zone.py           # 禁停区绘制及违停判断
├── view_stream.py            # 不做推理的视频源预览工具
├── yolo26x.pt                # 通用目标检测权重
├── license_plate_best.pt     # 车牌检测权重
├── requirements.txt          # 基础 Python 依赖
└── 使用说明.md               # 本文档
```

### 2.1 主要符号

| 文件 | 符号 | 职责 |
|---|---|---|
| `detect_stream.py` | `Config` | 保存模型、阈值、设备、窗口等全局配置 |
| `detect_stream.py` | `ThreadedCapture` | 网络流后台抓帧，只保留最近一帧 |
| `detect_stream.py` | `run()` | 启动完整阻塞式处理循环 |
| `mog_anomaly_engine.py` | `MOGAnomalyEngine` | 背景建模、候选跟踪、异常分类与告警 |
| `mog_anomaly_engine.py` | `AnomalyAlert` | MOG2 告警数据类 |
| `parking_zone.py` | `ParkingZoneDetector` | 禁停区域、车辆轨迹和违停状态 |
| `parking_zone.py` | `draw_zone_interactive()` | 使用鼠标绘制禁停多边形 |

### 2.2 当前帧处理流程

```text
摄像头/文件/网络流
        |
        +-- 网络流 -> ThreadedCapture -> 最近一帧
        +-- 文件/摄像头 -> cv2.VideoCapture
        |
        v
全帧车牌检测（both 模式）
        |
        v
车辆 YOLO 检测或 ByteTrack 跟踪
        |
        +-- 正常类别 -> 车辆框
        +-- 非白名单类别 -> YOLO 异常投票
        +-- 每个车辆裁剪区域 -> 车牌补检 -> 可选 OCR
        |
        +-- 跟踪结果 -> 禁停区判断
        +-- 车辆框 -> MOG2 正常车辆抑制
        |
        v
绘制检测框、告警、HUD
        |
        +-- GUI 窗口/截图
        +-- 无 GUI 视频写出
```

MOG2、ByteTrack、车牌平滑缓存和禁停轨迹都是有状态组件。生产集成时，每路视频应维护
独立状态，不能把同一份跟踪状态无隔离地混用于多个视频源。

## 4. 模型文件与必须修正的路径

### 4.1 本地权重

| 文件 | 实际文件大小 | 当前预期用途 |
|---|---:|---|
| `yolo26x.pt` | 118,667,365 字节，约 113.2 MiB | 通用目标及车辆检测 |
| `license_plate_best.pt` | 175,045,023 字节，约 166.9 MiB | 车牌检测 |

旧说明中车牌模型标记为约 43 MB，与当前文件不一致。模型架构、类别和精度应通过可信来源、
训练记录及实际推理验证，不要仅凭文件名判断。

### 4.2 当前阻断问题

`detect_stream.py` 中的车牌模型仍指向另一台计算机：

```python
PLATE_MODEL = "C:/Users/21495/runs/detect/license_plate_26m/weights/best.pt"
```

该路径在当前计算机不存在，因此以下模式会在模型加载阶段失败：

- 默认的 `--model both`。
- `--model plate`。
- 任何依赖车牌模型的 OCR 使用方式。

只有 `--model vehicle` 不受该车牌路径影响。

### 4.3 推荐修正

在 `detect_stream.py` 中让模型路径相对脚本目录解析：

```python
class Config:
    VEHICLE_MODEL = str(Path(__file__).with_name("yolo26x.pt"))
    PLATE_MODEL = str(Path(__file__).with_name("license_plate_best.pt"))
```

这比简单使用相对路径更可靠，因为服务进程的工作目录通常不等于脚本目录。

更适合生产系统的做法是新增 `--vehicle-model`、`--plate-model` 参数，或从配置文件和环境变量
读取模型路径。当前代码尚未实现这些配置入口。

### 4.4 权重安全

PyTorch `.pt` 文件可能包含基于 Pickle 的序列化内容。只加载来源可信、经过校验的模型权重；
不要允许外部用户上传任意 `.pt` 文件并由服务直接加载。生产发布时应记录文件哈希。

## 6. 命令行参数

| 参数 | 类型/取值 | 默认值 | 说明 |
|---|---|---:|---|
| `--source`, `-s` | 摄像头编号、路径或 URL | `0` | 视频输入；纯数字字符串会转为摄像头编号 |
| `--model`, `-m` | `vehicle` / `plate` / `both` | `both` | 要加载的检测模型 |
| `--conf` | 浮点数 | `0.4` | 车辆模型置信度阈值 |
| `--plate-conf` | 浮点数 | `0.20` | 全帧车牌检测置信度阈值 |
| `--skip` | 正整数 | `1` | 车辆检测及 plate-only 检测的帧间隔 |
| `--track` | 开关 | 关闭 | 使用 ByteTrack，启用跟踪 ID |
| `--ocr` | 开关 | 关闭 | 使用 HyperLPR3 识别车牌字符 |
| `--resize` | 整数 | `1280` | GUI 窗口目标宽度，不改变模型输入源分辨率 |
| `--output`, `-o` | 文件路径 | 无 | 仅无 GUI 分支使用的视频输出路径 |
| `--device`, `-d` | `cpu` / `cuda:0` 等 | `cuda:0` | Ultralytics 推理设备 |
| `--mog` | 开关 | 关闭 | 创建并启用 MOG2 异常引擎 |
| `--mog-debug` | 开关 | 关闭 | 打印 MOG2 调试信息并显示前景掩码 |
| `--parking` | 开关 | 关闭 | 创建禁停检测器 |
| `--parking-time` | 浮点数，秒 | `3.0` | 填满停留轨迹所需的目标时长 |

查看程序自身帮助：

```powershell
python .\detect_stream.py --help
```

### 6.1 `--model` 模式差异

| 模式 | 车辆模型 | 车牌模型 | 说明 |
|---|---:|---:|---|
| `vehicle` | 加载 | 不加载 | 适合基础车辆、YOLO 异常、MOG2 和禁停检测 |
| `plate` | 不加载 | 加载 | 只做全帧车牌检测，不能提供车辆跟踪 |
| `both` | 加载 | 加载 | 全帧车牌检测并在车辆裁剪区域补检 |

### 6.2 `--skip` 的实际边界

当前 `--skip` 不会统一降低所有模型调用频率：

- 它控制车辆模型推理。
- 它控制 `plate` 模式下的全帧车牌推理。
- 在 `both` 模式下，全帧车牌检测仍然每轮执行。
- 车辆裁剪区域的车牌补检也可能继续使用最近一次车辆结果执行。

因此不要假设 `--skip 2` 一定能把全部 GPU 负载降低一半。生产重构时应为每个推理阶段设置
独立采样频率。

## 8. 检测逻辑与调优

### 8.1 YOLO 正常类别与异常类别

`Config.NORMAL_CLASSES` 当前为：

```python
{0, 1, 2, 3, 5, 6, 7, 9, 10, 11}
```

按常见 COCO 类别映射，它包含人员、自行车、汽车、摩托车、公交车、火车、卡车、
交通灯、消防栓和停止标志。这里的“正常”不是严格的车辆集合，而是道路场景白名单。

白名单之外的 YOLO 类别会进入异常逻辑：

- 有跟踪 ID 时，至少累计 5 帧，并要求异常票数占比不低于 `0.6`。
- 确认后仍要求当前置信度不低于 `0.5`。
- 没有跟踪 ID 时，单帧置信度达到 `0.5` 即作为异常。

这只是基于 COCO 类别的规则，不等同于经过道路遗撒物数据训练的专用异常模型。

### 8.2 车牌检测

`both` 模式包含两条车牌检测路径：

1. 对整帧执行车牌模型，阈值 `0.20`。
2. 对每个车辆框增加约 20% 边距后裁剪补检，阈值 `0.12`。

补检会过滤尺寸过大和宽高比过方的候选框。车牌框消失后会保留约 5 个处理循环以减轻闪烁。

### 8.3 HyperLPR3 OCR

OCR 置信度阈值为 `0.5`。代码使用基于车牌框坐标的临时 ID 缓存识别文本，并采用出现次数最多
的结果。该 ID 不是 ByteTrack 车牌轨迹 ID，车辆或摄像机移动时可能发生变化。

生产接入建议：

- 将车牌结果关联到车辆 `track_id`。
- 保存原始识别置信度和候选列表。
- 对车牌文本进行格式校验，但不要用格式规则掩盖模型错误。
- 为车牌等个人信息设置访问控制、脱敏和保留期限。

### 8.4 MOG2 七层过滤

`MOGAnomalyEngine` 的处理包含：

1. 可选道路 ROI。
2. 轮廓面积过滤。
3. 持续时间和检测率过滤。
4. MOG2 阴影标记与 HSV 暗区过滤。
5. 开运算和闭运算去噪。
6. 与 YOLO 正常车辆框融合，抑制车辆告警。
7. 背景模型预热和重置能力。

主程序中的默认值：

| 参数 | 默认值 | 说明 |
|---|---:|---|
| `MOG_MIN_AREA` | `150` | 最小轮廓面积，像素平方 |
| `MOG_MIN_DURATION` | `2.0` | 普通候选的确认时长 |
| `MOG_MAX_DURATION` | `5.0` | 近期车辆区域使用的更长确认时长 |
| `MOG_WARMUP` | `100` | 背景模型预热帧数 |
| `var_threshold` | `25` | 引擎构造函数默认方差阈值 |
| `history` | `500` | 引擎构造函数默认历史长度 |

当前主程序没有调用 `set_roi()`，所以 ROI 层默认不生效，MOG2 会分析整幅画面。

MOG2 告警类型是按轮廓面积和宽高比划分的规则标签：

- `small_debris`
- `medium_object`
- `elongated_object`
- `large_object`

这些标签不是语义分类结果。雨雪、树影、曝光变化、摄像机抖动、OSD 和压缩噪声都可能造成误报。

### 8.5 禁停检测

禁停检测输入格式：

```python
(x1, y1, x2, y2, track_id)
```

判断过程：

1. 使用车辆框中心点判断是否进入任一禁停多边形。
2. 按 `track_id` 保存位置序列。
3. 当位置序列长度达到 `fps * parking_time` 后，比较首尾位移。
4. 位移小于 `30` 像素时产生违停结果。
5. 新 ID 距旧 ID 最近位置小于 `50` 像素时尝试继承轨迹。
6. 失联轨迹保留约 2 秒后清理。

依赖关系：

- 必须使用 `--track`，否则没有 `track_id`。
- 必须按 `Z` 添加至少一个区域。
- 输入 FPS 必须可信，否则基于帧数的停留窗口会偏离期望。
- 当前使用框中心点，不是车轮接地点或车辆与区域的相交面积。

## 9. Python 进程内集成

### 9.1 直接调用 `run()`

当前入口：

```python
run(
    source,
    mode="both",
    use_track=False,
    use_ocr=False,
    out_path=None,

```

最小调用示例：

```python
from pathlib import Path

import detect_stream

module_dir = Path(detect_stream.__file__).resolve().parent

detect_stream.Config.VEHICLE_MODEL = str(module_dir / "yolo26x.pt")
detect_stream.Config.PLATE_MODEL = str(module_dir / "license_plate_best.pt")

    mode="both",
    use_track=True,
    use_ocr=False,

```

该方式只适合快速嵌入或验证，原因如下：

| 限制 | 影响 |
|---|---|
| `run()` 是阻塞循环 | 会占用调用线程 |
| 结果只被画到帧或打印 | 业务代码无法稳定消费结构化结果 |
| 配置使用全局 `Config` 类变量 | 多实例配置会互相覆盖 |
| 导入时探测 OpenCV GUI | 服务进程可能产生 GUI 副作用 |
| 停止依赖按键、视频结束或中断 | 缺少外部 `stop_event` |
| 内部资源没有统一 `try/finally` | 异常中断时不保证完整释放 |

不要在 Web 请求处理线程中直接调用 `run()`。

### 9.2 直接复用 `MOGAnomalyEngine`

该类可以脱离主循环使用：

```python

    max_duration=5.0,
```

注意：

- `frame` 必须是 BGR `numpy.ndarray`。
- 建议传入尚未绘制框和 HUD 的原始帧副本。
- 每路视频创建独立的 `MOGAnomalyEngine`。
- 视角变化或切换视频源后调用 `reset()` 并重新预热。
- 可使用 `set_roi([(x, y), ...])` 限制道路区域。
- `snapshot` 是内存中的图像数组，主程序不会自动保存它。
- 方法类型注解、docstring 和主程序传入元组长度存在不一致；当前逻辑只读取前四项坐标。

### 9.4 `ThreadedCapture` 的可复用性

`ThreadedCapture` 适用于低延迟网络流原型：

```python
capture = ThreadedCapture(source)
try:
    ok, frame = capture.read()
finally:
    capture.release()
```

但是当前实现没有帧序号、采集时间戳、帧新鲜度判断或真正的重新打开逻辑。业务循环可能多次读到
同一最新帧。生产系统应返回类似以下结构：

```text
FramePacket(stream_id, sequence, captured_at, frame)
```

并只对新的 `sequence` 执行一次推理。

### 9.5 推荐的进程内接口

```python
class VideoAnalyticsPipeline:

        """返回 list[DetectionEvent]，不负责 GUI、网络或数据库。"""
```

推荐把以下职责从 `run()` 分离：

- `Renderer`：可选画框。

## 10. 独立推理服务集成

本节描述推荐的生产架构，不代表当前目录已经实现服务接口。

### 10.1 推荐架构

```text
业务系统
   |
   +-- 流管理 API ------------------------------+
   |                                           |
   v                                           v
Stream Manager                           Health / Metrics
   |
   +-- Capture Worker（每路流）
          |
          v
      Latest Frame Queue（容量 1 或很小的有界队列）
          |
          v
      Inference Scheduler
          |
          v
      VideoAnalyticsPipeline
        |      |       |        |
      YOLO   Plate    MOG2    Parking
          |
          +-- DetectionEvent
                  |
                  +-- 消息队列 / WebSocket / SSE
                  +-- 事件数据库
                  +-- 快照或短视频存储
```

### 10.2 组件边界

| 组件 | 职责 | 不应承担 |
|---|---|---|
| Stream Manager | 创建、停止、查询流任务 | 模型推理 |
| Capture Worker | 打开视频、采集、时间戳、重连 | 业务告警 |
| Frame Queue | 背压和丢弃旧帧 | 无限缓存 |
| Inference Scheduler | GPU 调度、批处理、并发限制 | 保存 RTSP 凭据 |
| Analytics Pipeline | 检测、跟踪、规则、结构化结果 | HTTP 会话管理 |
| Event Publisher | 重试、幂等、发布事件 | 修改检测结果 |
| Snapshot Store | 保存快照、生命周期管理 | 长期保存所有原始帧 |

### 10.3 建议的管理接口

以下接口名称仅为设计示例：

| 方法 | 路径 | 用途 |
|---|---|---|
| `POST` | `/v1/streams` | 创建并启动一路分析任务 |
| `GET` | `/v1/streams/{stream_id}` | 查询连接、FPS、延迟和错误状态 |
| `DELETE` | `/v1/streams/{stream_id}` | 停止任务并释放资源 |
| `POST` | `/v1/streams/{stream_id}/zones` | 更新禁停区和 MOG ROI |
| `GET` | `/health/live` | 进程存活检查 |
| `GET` | `/health/ready` | 模型和 GPU 就绪检查 |
| `GET` | `/metrics` | 暴露监控指标 |

检测结果更适合通过消息队列、WebSocket 或 SSE 发布，而不是让长时间推理占用同步 HTTP 请求。

## 11. 并发、生命周期与资源管理

### 11.1 单路视频

推荐每路流具有独立的：

- 采集状态和重连计数。
- ByteTrack 状态。
- MOG2 背景模型。
- 禁停区域和停车轨迹。
- OCR 投票缓存。
- 帧序号和最后成功采集时间。

### 11.2 多路视频

不要直接让多个线程并发调用同一个带 `persist=True` 的 YOLO 跟踪实例。跟踪状态可能混流，
并且模型对象的并发安全性不应在未经验证时假定成立。

可选部署方式：

1. 每路流独立进程：隔离最好，显存占用较高。
2. 每个 GPU 一个推理进程：模型共享，由调度器串行或批量推理；每路跟踪状态独立。
3. 算法服务池：适合大量流，但需要明确会话和跟踪状态归属。

### 11.3 背压策略

实时监控通常优先低延迟，而不是处理每一帧。建议：

- 采集队列容量设为 1 或很小的固定值。
- 推理落后时丢弃旧帧，保留最新帧。
- 录像归档使用独立通道，不与实时推理共用丢帧队列。
- 记录采集时间到推理完成时间的端到端延迟。

离线视频分析则通常要求不丢帧，应使用不同的队列和任务模型。

### 11.4 优雅停止

生产循环应使用 `stop_event` 和 `try/finally`：

```text
设置停止信号
  -> 停止读取新帧
  -> 等待或取消推理
  -> flush 事件
  -> release VideoCapture/VideoWriter
  -> 清理跟踪与背景状态
  -> 标记任务 stopped
```

当前 `run()` 没有外部停止句柄，也没有覆盖整个循环的 `try/finally`。

## 12. 性能和延迟调优

### 12.1 先测量再调参

至少记录：

- 采集 FPS。
- 实际推理 FPS。
- 每个模型的推理耗时。
- 队列长度和丢帧数。
- 端到端帧延迟。
- GPU 利用率、显存、温度。
- 每路流重连次数。
- 每类告警数量和抑制数量。

### 12.2 常用调优方向

| 目标 | 优先措施 | 代价 |
|---|---|---|
| 降低延迟 | 只保留最新帧、减小缓冲 | 会丢帧 |
| 提高吞吐 | 使用较小模型、降低输入尺寸 | 可能降低召回率 |
| 降低显存 | 减少模型副本、限制并发 | 调度更复杂 |
| 降低 OCR 开销 | 只对稳定高质量车牌 OCR | 文字结果出现更晚 |
| 降低 MOG 误报 | 设置道路 ROI、延长持续时间 | 小物体告警变慢 |
| 降低车牌闪烁 | 增加平滑或跟踪关联 | 可能保留过期框 |

模型文件名替换为更小规格并不保证兼容。更换权重后必须验证类别映射、输入尺寸、准确率和速度。

### 12.3 分辨率

`--resize` 只改变 GUI 窗口尺寸，不会主动缩小传入 YOLO 的原始帧。要降低模型输入成本，
应在推理管线中显式设置 Ultralytics 输入尺寸或缩放帧，并正确把坐标映射回原始画面。

### 12.4 MOG2 时间基准

MOG2 的持续时间使用 `time.time()`，不是视频文件的媒体时间戳。离线视频如果以快于或慢于
原始帧率的速度处理，确认时长会随实际处理速度变化。生产重构应把可信的帧采集时间或视频 PTS
传给 `process(timestamp=...)`。

禁停判定主要按 `fps * parking_time` 的帧数量建立窗口，但返回的 `duration` 又使用墙上时间。
离线处理和不准确的网络流 FPS 都可能导致语义不一致。

## 13. 可靠性、可观测性与安全

### 13.1 网络流恢复

代码中虽然存在 `RTSP_REOPEN_TIMEOUT` 和连接状态字段，但当前抓帧线程在读取失败后只是继续
调用原有 `VideoCapture.read()`，没有关闭并重新打开流。`last_reconnect` 也没有形成完整重连流程。

生产实现至少需要：

- 连续失败阈值。
- 带抖动的指数退避。
- 主动 `release()` 后重新创建 `VideoCapture`。
- 最大重试频率和熔断。
- 最后新鲜帧时间，而不只是“曾经读到过帧”。
- 重连后重置 ByteTrack、MOG2 和帧时间基准。

### 13.2 建议日志字段

日志应结构化并包含：

```text
timestamp, level, stream_id, component, frame_id,
track_id, event_type, latency_ms, retry_count, error_code
```

不要在日志中打印含用户名和密码的完整 RTSP URL。

### 13.3 建议指标

- `stream_connected`
- `last_frame_age_seconds`
- `capture_fps`
- `inference_fps`
- `inference_latency_seconds`
- `frame_queue_depth`
- `frames_dropped_total`
- `stream_reconnects_total`
- `events_total{type=...}`
- `gpu_memory_bytes`

### 13.4 安全与合规

- 将 RTSP 凭据放入密钥管理系统，不写入源码、命令历史和普通日志。
- 限制服务可访问的视频地址，防止利用 URL 进行内网探测。
- 只加载可信并校验哈希的模型文件。
- 对车牌文字、截图和录像设置访问控制、审计、加密和保留期限。
- 核查 Ultralytics、HyperLPR3、OpenCV、模型权重和训练数据的许可证及商业使用条件。
- API 应使用鉴权、速率限制和输入校验。

## 15. 测试建议

当前目录没有测试文件。集成前建议建立以下测试层次。

## 16. 常见问题排查

| 现象 | 可能原因 | 处理 |
|---|---|---|
| 启动时报车牌权重不存在 | `PLATE_MODEL` 是其他机器绝对路径 | 按第 4 节修改路径 |
| `CUDA` 或 device 错误 | PyTorch CUDA 不可用或设备编号错误 | 检查 `torch.cuda.is_available()`，或使用 `--device cpu` |
| `Cannot open source` | 地址、权限、协议或编解码问题 | 先用 `view_stream.py` 验证，再检查 URL 和网络 |
| RTSP 延迟越来越大 | 解码、网络、缓冲或推理跟不上 | 使用最新帧队列并测量端到端延迟 |
| 断流后不能恢复 | 当前没有完整 reopen 逻辑 | 实现显式释放、退避和重新打开 |
| 没有停车告警 | 未启用跟踪、未画区域或车辆未静止足够久 | 使用 `--track --parking`，按 `Z` 设置区域 |
| OCR 无结果 | 未安装 HyperLPR3、车牌太小或置信度不足 | 检查依赖、车牌框和原始裁剪质量 |
| MOG2 初期大量前景 | 背景尚未预热 | 等待至少 100 个处理循环 |
| MOG2 误报阴影或 OSD | 未设置 ROI，环境变化明显 | 设置 ROI、遮蔽 OSD、调整面积和持续时间 |
| 出现 `ANOMALY:...` | YOLO 类别不在白名单 | 核对类别映射并按业务调整白名单 |
| 传入 `--output` 没有文件 | 当前处于 GUI 分支 | 参考第 7.5 节重构输出开关 |
| 无显示服务器报 OpenCV GUI 错误 | 使用 GUI 版 OpenCV 或调用了 `imshow` | 使用 headless 部署方案并移除 GUI 调用 |
| FPS 低 | 模型较大、同时运行两个模型和 OCR | 分项测量，降低输入或换经验证的小模型 |
| 车牌框闪烁 | 单帧召回不稳定 | 调整阈值、改用车牌跟踪和多帧融合 |

## 17. 推荐实施顺序

1. 使用 `view_stream.py` 验证目标视频源。
2. 用 `--model vehicle` 验证车辆权重、CUDA 和基础性能。
3. 修正车牌模型路径，分别验证 `plate` 和 `both`。
4. 使用固定视频建立当前准确率和性能基线。
5. 从 `run()` 提取无 GUI 的 `process_frame()` 和结构化事件。
6. 将配置从全局类变量改为每路流实例配置。
7. 增加流生命周期、时间戳、重连、停止和资源释放。
8. 接入道路 ROI、禁停区域持久化和业务事件发布。
9. 增加指标、结构化日志、健康检查和告警。
10. 完成多路压力测试、故障演练、安全评审和许可证核查。
