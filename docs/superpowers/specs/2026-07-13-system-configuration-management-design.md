# 系统配置管理设计

- 日期：2026-07-13
- 项目：VideoTest / 沙盘交通智控台
- 状态：设计已确认，等待正式文档复核
- 存储方案：SQLite 配置仓库 + 内容寻址资源目录

## 1. 背景

当前系统已经具备 RTSP 视频选择、道路拓扑编辑、禁停场景、道路异常场景、白名单和检测参数等能力，但这些配置分散在代码常量、多个 JSON 文件和进程内状态中：

- `backend/config.py` 中的 `STREAM_SOURCES` 固定保存 12 路 RTSP 地址。
- `traffic_map.json` 同时保存一份道路几何、一张底图路径和 12 个摄像头位置。
- `runtime/no_parking/scenes.json` 与 `runtime/road_abnormal/scenes.json` 分别保存两类场景档案。
- `whitelist.json` 保存车辆白名单。
- 检测设备、阈值和抽帧间隔主要是运行时配置，缺少统一持久化。
- 当前禁停与道路异常共享一个主 `VideoStreamService`，不能在不同摄像头上并行运行。

这些边界可以支撑单一演示配置，但无法可靠支持多方案管理、独立切换、完整迁移和失败回滚。本设计新增系统管理页与配置中心，在不重写现有检测算法的前提下，把所有业务配置纳入统一的数据模型和操作协议。

## 2. 目标

本期实现以下能力：

1. 固定当前 12 个摄像头为全局主数据，摄像头 ID 永久稳定。
2. 管理任意数量的 RTSP 流资源，允许先保存离线流并单独执行连通性测试。
3. 管理多套摄像头-RTSP 关联方案，每套方案完整、一对一地映射 12 个摄像头与 12 路唯一流，并支持预检后一键切换。
4. 管理多套道路拓扑方案。每套方案拥有独立底图、道路几何、连通节点、道路方向以及 12 个摄像头的位置、朝向、覆盖范围和道路绑定。
5. 独立启用摄像头-流关联方案、道路拓扑方案、禁停场景和道路异常场景。
6. 禁停与道路异常各自全局最多启用一个场景，并允许两类场景绑定不同摄像头并行运行。
7. 通过可移植 ZIP 协议导出和全量导入所有业务配置及其资源文件。
8. 导入时恢复包内激活状态，失败时回滚到导入前配置。
9. 在可信单用户模式下提供危险操作确认和本地审计，不引入登录系统。
10. 保持现有监控、拓扑、场景和白名单 API 的兼容性。

## 3. 非目标

以下内容不在本期范围内：

- 登录、RBAC、多人协作和远程配置审批。
- MySQL、分布式配置中心和多进程配置一致性。
- 每种场景同时运行多个档案。
- 单个场景同时分析多个摄像头。
- 摄像头透视标定、单应性矩阵、内参和外参管理。
- RTMP、HTTP-FLV、ONVIF 等 RTSP 以外的新接入协议。
- 告警事件、事件截图、日志、上传视频和运行统计的导入导出。
- `.env`、监听端口、本机模型路径和模型文件的导入导出。
- 配置包加密、签名和凭据脱敏。

## 4. 原始需求中补充的必要能力

为了让原始三项功能形成可靠闭环，本设计补充以下能力：

- 固定摄像头主数据，避免继续混用摄像头 ID 与流 ID。
- 三个配置域的独立激活指针与跨域兼容性校验。
- RTSP 并行预检、运行通道重连和失败补偿。
- 道路几何与真实连通图的组合模型。
- 拓扑修订号和场景失效规则。
- 底图、参考图等资源文件的内容摘要和受控路径。
- 配置包协议版本、完整性校验、旧版迁移和安全解包。
- 全量导入预览、自动备份、原子替换和崩溃恢复。
- 内置只读基线、引用保护和受控删除。
- 操作状态、审计记录和稳定错误码。
- 现有配置的一次性迁移与旧 API 兼容层。

## 5. 已确认的产品决策

### 5.1 固定摄像头

- 摄像头目录固定为当前 12 个摄像头。
- `camera_id` 是跨流方案、拓扑方案和场景档案的稳定身份。
- 系统管理页不能新增、删除或修改 `camera_id`。
- 显示名称和排序作为内置数据随应用发布。

### 5.2 独立激活

以下状态可以分别切换：

- 当前摄像头-流关联方案。
- 当前道路拓扑方案及其修订号。
- 当前禁停场景，可为空。
- 当前道路异常场景，可为空。

不引入把四者强行捆绑为一个总方案的系统快照。

### 5.3 流关联规则

- 非激活方案可以作为草稿存在并暂时不完整。
- 激活前必须恰好覆盖全部 12 个摄像头。
- 同一方案内同一路流只能绑定一个摄像头。
- 新增流可以在离线状态下保存。
- 激活方案前必须并行验证 12 路流，并取得可解码首帧。
- 内置默认关联方案只读且不可删除，修改时必须复制为新方案。

### 5.4 拓扑规则

- 每个拓扑方案拥有独立底图和独立坐标空间。
- 道路同时保存可视几何与图连通关系。
- 摄像头参数仅包含 `x`、`y`、`heading`、`view_range` 和绑定路段，不增加透视标定。
- 非内置拓扑允许原地修改，每次成功保存都递增 `revision`。
- 内置默认拓扑只读且不可删除，修改时必须复制为新方案。

### 5.5 场景规则

- 场景档案绑定 `topology_id + topology_revision + camera_id`。
- 禁停和道路异常各自全局最多启用一个场景。
- 两类场景可以绑定不同摄像头并行运行。
- 切换到另一拓扑时，所有与新激活拓扑不兼容的运行场景自动停用，但原档案仍保持有效，可在切回对应拓扑后重新启用。
- 直接修改一个拓扑并递增修订号时，绑定旧修订号的场景进入 `needs_review` 状态，不能再次启用，直到用户打开场景并重新确认、保存到当前修订号。

### 5.6 导入导出规则

- 导出格式为包含 JSON 和资源文件的 ZIP，而不是单一 JSON。
- RTSP URL 及其中的用户名、密码或令牌按用户选择原样明文导出。
- 导入始终执行用户配置的全量替换，不提供合并模式。
- 导入前必须完成预检和影响预览，并由用户再次确认。
- 导入后恢复包内激活状态。
- 目标系统内置默认方案始终保留，导入包不能覆盖或删除。
- 导入失败时恢复自动备份。

### 5.7 安全边界

- 系统维持可信单用户模式，不增加登录。
- 导入、导出、切换、删除和编辑激活方案需要明确确认。
- 所有配置变更写入本地审计。
- UI 默认遮挡 RTSP URL 中的 userinfo，但导出包仍保存明文。
- 审计记录不得包含 RTSP 凭据明文。

## 6. 总体架构

系统分为四层：

1. 系统管理前端。
2. 配置 API 与应用服务。
3. SQLite 配置仓库和资源目录。
4. 现有运行时服务的适配层。

### 6.1 新增应用服务

#### ConfigurationService

职责：

- 配置 CRUD。
- Pydantic 请求校验。
- 外键、唯一性和业务引用校验。
- 内置基线保护。
- 拓扑修订递增和场景失效标记。
- 受控删除与依赖清单。

#### ActivationCoordinator

职责：

- 串行化导入、切换和激活写操作。
- 执行预检。
- 登记进行中的配置操作。
- 把新配置应用到运行时。
- 更新激活指针和审计。
- 失败时恢复旧运行态。

#### ImportExportService

职责：

- 创建一致的配置快照。
- 生成 ZIP 和文件摘要。
- 安全解包、协议校验和旧版迁移。
- 在临时数据库中执行全量预检。
- 生成导入预览与一次性确认令牌。
- 自动备份、全量替换和失败恢复。

#### StreamProbeService

职责：

- 使用受限超时并行探测 RTSP 流。
- 区分 DNS、TCP、RTSP 握手、认证、超时和不可解码等失败。
- 返回每路流的结构化结果。
- 不把一次成功探测等同于长期在线状态。

#### ConfigurationRepository

职责：

- 使用 Python 标准库 `sqlite3` 封装持久化。
- 开启 `PRAGMA foreign_keys = ON`、WAL 和合理的 busy timeout。
- 提供显式事务和迁移版本。
- 隐藏 SQL 与 SQLite 文件布局，使应用服务不直接操作表。

配置变更锁是进程内锁，因此本期部署必须使用单个 Uvicorn worker。多 worker 或多后端实例必须在后续项目中引入跨进程锁和配置通知后才能启用。

#### AssetStore

职责：

- 在受控目录保存拓扑底图和场景参考图。
- 以 SHA-256 内容摘要命名，文件不可原地修改。
- 校验媒体类型、图片可解码性、尺寸和文件大小。
- 仅在数据库无引用后执行延迟清理。

### 6.2 运行时适配

`ApplicationState` 继续作为现有服务的组合入口，但配置来源改为仓库和激活状态。运行时至少包含：

- 主监控通道。
- 道路热力分析通道。
- 禁停场景独立分析通道。
- 道路异常独立分析通道。
- 多摄像头预览服务。

所有消费者只持有 `camera_id`，通过当前激活流关联方案解析实际 RTSP URL。拓扑和场景逻辑不直接保存流 URL。

### 6.3 文件布局

建议使用：

```text
runtime/config/
  config.sqlite3
  assets/
    maps/
    scene-references/
  backups/
  staging/
```

数据库是业务配置的唯一事实来源。资源目录只保存由 `asset` 表引用的不可变文件。

- `error_summary`
- `started_at`

#### schema_metadata

保存 SQLite schema 版本、首次迁移标记和内置基线版本。

### 7.2 删除规则

- 内置记录不能删除。
- 激活中的方案或场景不能删除。
- 仍被其他配置引用的流、拓扑或资源不能删除。
- API 返回依赖实体列表，不执行隐式级联删除。
- 资源文件只有在数据库提交后确认无引用时才进入垃圾清理。

## 8. 激活与一致性

SQLite 事务不能覆盖 RTSP 网络 I/O 和运行线程，因此采用应用级协调协议，不在探测流时长时间持有数据库写事务。

### 8.1 通用六步流程

1. 获取进程内配置变更锁。
2. 完成所有结构、资源、引用和网络预检，不改变当前激活状态。
3. 短事务写入 `configuration_operation = applying`，激活指针仍指向旧配置。
4. 将目标配置应用到运行时，并等待必要的解码首帧或服务 ready。
5. 成功时事务更新 `activation_state` 和审计；失败时恢复旧运行态并记录失败。
6. 释放锁并返回逐项结果、耗时、自动停用项和回滚状态。

启动时以最后已提交的 `activation_state` 为唯一事实来源，重新装载所有运行通道。遗留的 `applying` 操作标记为 `interrupted`。

### 8.2 流关联方案激活

预检：

- 方案包含 12 个不同摄像头。
- 方案包含 12 路不同、已启用的流。
- 所有流通过并行 RTSP 探测并取得可解码首帧。

运行时应用：

- 替换 `camera_id -> rtsp_url` 解析器。
- 只重连当前正在消费且 URL 发生变化的通道。
- 多摄像头预览服务同步使用新解析器。
- 任一需要重连的通道失败时恢复全部旧映射。

编辑当前激活的普通流方案视为一次重新激活。请求必须提交完整映射，成功前旧数据库内容和旧运行映射保持有效。

### 8.3 拓扑方案激活

预检：

- 底图存在且摘要正确。
- 包含 12 个摄像头。
- 道路几何、节点、起止节点、方向和外键均合法。

运行时应用：

- 在 `map_lock` 下替换 `TrafficMapModel`。
- 清空轨迹、热度等瞬时状态。
- 停止并清空与新拓扑不兼容的激活场景。
- 返回被自动停用的场景清单。

### 8.4 修改当前拓扑

修改普通拓扑时，路段、节点、摄像头和底图作为一笔事务保存，并递增 `revision`。随后：

- 当前 `activation_state.topology_revision` 更新到新修订号。
- 绑定旧修订号的全部场景标记为 `needs_review`。
- 正在运行的旧修订场景自动停用。
- 用户必须在对应场景编辑器中重新确认区域和摄像头关系，再保存为当前修订号。

### 8.5 场景激活

预检：

- 场景为 `ready`。
- 场景绑定的拓扑 ID 和修订号与当前激活拓扑一致。
- 摄像头存在于当前拓扑和当前流关联方案。
- 参考图和参数合法。
- 对应流可以连接并解码。

禁停与道路异常分别使用独立 `VideoStreamService`。切换其中一个场景时，只启动、重连或停止对应类型的运行通道，不影响另一类型。

## 9. 配置包协议

### 9.1 协议身份

- 格式名：`videotest.configuration`
- 初始 `schema_version`：1
- 容器：ZIP
- 字符集：UTF-8
- JSON 对象拒绝未知字段。

### 9.2 目录结构

```text
videotest-config.zip
├─ manifest.json
├─ config/
│  ├─ fixed-cameras.json
│  ├─ stream-sources.json
│  ├─ stream-binding-profiles.json
│  ├─ topology-profiles.json
│  ├─ scene-archives.json
│  ├─ detection-settings.json
│  ├─ whitelist.json
│  └─ activation-state.json
└─ assets/
   ├─ maps/{sha256}.{ext}
   └─ scene-references/{sha256}.{ext}
```

领域 JSON 保存用户管理的数据。内置默认方案只通过稳定 ID 和指纹描述，不把其可覆盖内容作为导入数据。`fixed-cameras.json` 用于兼容性验证，不覆盖目标系统摄像头目录。

### 9.3 manifest.json

必备字段：

- `format`
- `schema_version`
- `export_id`
- `exported_at`
- `app_version`
- `contains_plaintext_credentials: true`
- `camera_catalog_fingerprint`
- `builtin_baseline_fingerprint`
- 各领域实体计数
- 每个文件的相对路径、SHA-256、字节数和媒体类型

SHA-256 只提供完整性检查，不提供来源认证或防篡改签名。

### 9.4 导出范围

包含：

- 固定摄像头兼容性描述。
- 所有用户流和流关联方案。
- 所有用户拓扑和独立底图。
- 所有场景档案和参考图。
- 白名单和全局启用状态。
- 检测参数。
- 当前激活状态。

流的最近探测结果和探测时间属于观测状态，不进入配置包；导入预检会重新生成这些状态。导出在配置变更锁下取得一个短暂的一致性快照，随后释放锁并流式生成 ZIP。内容寻址资源不可变，因此压缩阶段不需要持续阻塞配置写入。

排除：

- 告警事件、事件截图、日志、上传视频和运行统计。
- `.env`、端口、本机路径和模型文件。
- 审计历史和自动备份历史。
- 目标系统内置基线的可覆盖内容。

### 9.5 安全限制

初始限制：

- ZIP 压缩后最大 256 MiB。
- 解压后最大 512 MiB。
- 最多 1000 个文件条目。
- 单张图片最大 50 MiB，最大尺寸 10000 x 10000。
- 禁止绝对路径、`..` 路径穿越、符号链接、重复条目和未在 manifest 中声明的文件。
- 下载响应设置 `Cache-Control: no-store`。

### 9.6 导入流程

1. 将上传包写入受控暂存目录。
2. 执行安全解包和 manifest 完整性校验。
3. 解析所有 JSON，并按 `schema_version` 执行显式迁移器。
4. 在临时 SQLite 中导入目标系统内置基线和包内用户数据。
5. 校验全部外键、唯一性、资源、拓扑、场景和激活状态。
6. 并行探测包内激活流方案的 12 路流，并验证检测设备可用。
7. 返回预览报告和一次性确认令牌。报告列出替换、删除、保留的数量，基线差异、流结果和目标激活状态。
8. 用户确认后获取配置变更锁，并自动导出当前配置到备份目录。
9. 在一个 SQLite 事务中保留目标内置基线、schema 元数据、操作日志和审计历史，删除并替换全部用户配置，安装缺失的内容寻址资源，并恢复包内激活状态。
10. 重建运行通道。进程内应用失败时恢复自动备份。

第 9 步的 SQLite 提交是全量导入的持久化提交点。提交前进程崩溃时旧配置保持有效；提交后进程崩溃时，下一次启动继续装载已导入配置并把遗留操作标记为 `interrupted`。如果进程仍存活但第 10 步运行时应用失败，则立即从自动备份恢复数据库和运行态。旧资源在备份保留期内不删除，之后只清理确认无数据库引用的文件。

一次性确认令牌绑定上传包摘要和预检结果，只能使用一次，并在 15 分钟后过期。

自动备份保留最近 5 份。备份同样包含明文 RTSP 凭据，只能位于本机配置目录。

### 9.7 版本兼容

- 包版本高于当前支持版本时拒绝导入。
- 旧版协议只有存在显式且已测试的迁移器时才允许导入。
- 固定 12 个 `camera_id` 集合不一致时拒绝导入，不按名称猜测映射。
- 内置基线指纹不一致时显示警告并保留目标基线。
- 目标检测设备不可用时预检失败，不静默降级。

流预检、激活和导入等可能超过普通请求时限的操作返回 `202 Accepted` 和 `operation_id`，前端轮询操作接口。

### 10.7 错误契约

错误响应统一包含：

```json
{
  "error": {
    "code": "STREAM_PROBE_FAILED",
    "message": "关联方案中有 2 路流未通过预检",
    "operation_id": "op_20260713_0001",
    "details": [],
    "rollback": "not_required"
  }
}
```

规则：

- 并发配置写操作返回 `409 CONFIG_MUTATION_IN_PROGRESS`。
- 请求字段或引用错误返回 `422` 和具体字段或实体 ID。
- 网络或运行时失败返回稳定业务错误码和逐流结果。
- 重复激活当前方案返回成功的无操作结果。
- 所有失败明确返回 `rollback` 为 `not_required`、`succeeded` 或 `failed`。

## 12. 一次性迁移

### 12.1 迁移来源

- `STREAM_SOURCES`
- `traffic_map.json`
- `whitelist.json`
- `runtime/no_parking/scenes.json`
- `runtime/no_parking/references/`
- `runtime/road_abnormal/scenes.json`
- `runtime/road_abnormal/references/`
- 当前检测参数默认值

### 12.2 迁移步骤

1. 检测 `schema_metadata` 是否已记录迁移完成，避免重复执行。
2. 在暂存目录创建新 SQLite，而不是直接修改正式文件。
3. 写入固定 12 摄像头。
4. 将当前 12 路流和关联关系写为只读默认关联方案。
5. 将当前 `traffic_map.json` 和底图写为只读默认拓扑方案，`revision = 1`。
6. 导入白名单和全局启用状态。
7. 导入两类场景档案并复制参考图，但不激活场景，因为当前系统没有可靠持久化场景激活状态。
8. 写入检测参数默认值。
9. 校验完整数据库和资源摘要。
10. 校验通过后原子安装数据库，并激活默认流关联和默认拓扑。

迁移失败时不替换正式数据库，不删除或修改旧文件，并输出可操作错误。新版本在这种情况下拒绝启动配置服务，不自动进入双写或旧存储兼容模式；管理员修复迁移错误后重试，或回退到仍使用旧文件的应用版本。旧文件在迁移成功后仍保留为只读来源，但新系统不再写入。

### 12.3 现有 API 兼容

- `/api/system` 增加配置能力和当前激活摘要，不删除原字段。
- 现有流选择接口接收 `camera_id`，由激活方案解析 URL。
- 现有地图接口默认操作当前激活拓扑；拓扑编辑页面通过显式 `topology_id` 操作目标方案。
- 现有场景接口内部改用配置仓库。
- 现有白名单接口内部改用配置仓库。
- 旧 JSON 在迁移后不再作为写入目标。

## 14. 验收标准

1. 首次升级后，当前 12 路关联关系和当前道路拓扑作为只读默认方案可见且已激活。
2. 管理员可以保存离线 RTSP 流，但包含离线流的方案不能激活。
3. 关联方案不完整或重复使用同一路流时不能激活。
4. 12 路流全部通过预检后，关联方案可以一键切换；运行通道失败时旧方案保持有效。
5. 每个拓扑方案拥有独立底图、道路几何、连通节点和 12 个摄像头信息。
6. 切换拓扑会清空瞬时地图状态并停用不兼容场景。
7. 修改拓扑会递增修订号，旧修订场景进入待确认状态。
8. 禁停和道路异常可以绑定不同摄像头并行运行，各自最多一个场景。
9. 配置包包含全部约定业务配置、底图和场景参考图，且明确标注包含明文凭据。
10. 导入预检可以列出替换、删除、保留和失败项；确认后全量替换并恢复包内激活状态。
11. 导入失败时自动恢复原配置，重启后仍使用最后已提交的激活状态。
12. 目标系统内置基线不能通过编辑、删除或导入被覆盖。
13. 所有危险操作写入审计，审计中不出现 RTSP 凭据。
14. 现有监控、地图、禁停、道路异常和白名单核心 API 测试继续通过。

## 15. 实施分解建议

该设计跨越存储、运行态、协议和前端，实施计划应拆为可独立验证的阶段：

1. SQLite 仓库、资源存储和一次性迁移。
2. RTSP 流目录、关联方案和激活协调器。
3. 多拓扑数据模型、修订规则和现有地图编辑器适配。
4. 场景仓库化和两个独立场景运行通道。
5. 配置包导入导出、自动备份和崩溃恢复。
6. 系统管理页面、审计和完整回归测试。

每个阶段必须保持现有 API 可运行，并在进入下一阶段前通过对应迁移和回滚测试。

## 16. 风险与控制

| 风险 | 控制措施 |
| --- | --- |
| 明文 RTSP 凭据泄露 | 导出二次确认、URL 默认遮挡、响应不缓存、审计脱敏、本地备份限量 |
| 12 路预检耗时 | 并行探测、单路超时、逐流进度和可取消操作 |
| 数据库与运行时中间状态 | 操作日志、激活指针最后提交、启动时按数据库收敛 |
| 拓扑修改使场景区域失真 | 自动递增 revision、场景 needs_review、禁止旧修订场景启用 |
| 全量导入删除本机数据 | 强制预览、二次确认、导入前自动备份、失败回滚 |
| 配置包恶意 ZIP | 严格限额、路径检查、摘要校验、未知文件拒绝、图片解码验证 |
| 独立场景通道增加 GPU/CPU 负载 | 保持每种场景最多一路、延续抽帧间隔、在运行状态中展示资源错误 |
| 旧 API 与新仓库不一致 | 所有接口统一经过应用服务，旧 JSON 迁移后停止写入 |

本设计不把 SHA-256、可信单用户模式或二次确认视为真正的访问控制。若系统从可信本机扩展到局域网多人使用，登录、权限、CSRF 防护和加密凭据应作为独立项目设计。
