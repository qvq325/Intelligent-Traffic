# 当前拓扑道路方案备份与恢复设计

## 目标

为当前版本的 `VideoTest` 项目制作一份完整、可校验、可回滚的拓扑道路方案备份。所有交付文件均位于 `data-store-copy`，恢复时不依赖网络或仓库历史。

## 已确认的当前状态

- 项目通过根目录 `traffic_map.json` 持久化拓扑方案。
- 当前数据格式版本为 `3`，包含 13 条道路和 12 个摄像头点位/道路绑定。
- `traffic_map.json` 引用底图 `sandpan/沙盘平面图1.png`，完整恢复必须同时包含该文件。
- 车辆轨迹、流量、占用率和热力值仅存在于运行时内存，不属于持久化方案，也不进入本次备份。
- GitNexus 对 `TrafficMapModel` 的上游影响分析为 LOW：3 个直接依赖、5 个受影响符号；恢复数据主要影响应用启动及地图加载。
- 本任务不修改任何现有函数、类或方法。

## 交付结构

```text
data-store-copy/
|-- DESIGN.md
|-- README.md
|-- MANIFEST.json
|-- restore.ps1
`-- backup/
    |-- traffic_map.json
    `-- sandpan/
        `-- 沙盘平面图1.png
```

`backup` 保留项目内的相对目录结构，使 `traffic_map.json` 中的底图路径在恢复后仍然有效。

## 备份流程

1. 解析源 `traffic_map.json`，确认它是有效 JSON，版本为 `3`，并记录道路和摄像头数量。
2. 解析 `map_image`，确认引用路径位于项目目录内且目标文件存在。
3. 复制拓扑 JSON 和底图到 `data-store-copy/backup` 对应位置。
4. 对源文件和备份文件计算 SHA-256；如果复制期间源文件发生变化或任一校验值不一致，则重新捕获快照。
5. 生成 `MANIFEST.json`，记录备份时间、源提交、拓扑统计、相对路径、文件大小及 SHA-256。

双重哈希和 JSON 解析用于避免在服务恰好写入配置时捕获到不完整快照，无需为了备份主动中断当前服务。

## 恢复流程

`README.md` 提供逐条手工恢复命令，`restore.ps1` 提供同等操作的自动化入口。恢复流程如下：

1. 使用项目现有 `stop.ps1` 停止服务，避免恢复期间应用再次保存旧的内存状态。
2. 在修改项目文件前，校验备份文件是否与 `MANIFEST.json` 完全一致，并确认拓扑版本为 `3`。
3. 将项目当前的 `traffic_map.json` 和当前底图保存到 `data-store-copy/restore-rollback/<时间戳>`。
4. 将备份 JSON 和底图恢复到项目根目录的对应相对路径。
5. 再次校验目标文件 SHA-256、JSON 版本、道路数量、摄像头数量及底图存在性。
6. 使用项目现有 `start.ps1` 启动服务，通过健康接口、地图接口和底图接口做运行验证。

脚本默认以 `data-store-copy` 的父目录作为项目根目录，并支持显式传入其他项目根目录，以便在临时目录中演练恢复。脚本在全部备份校验通过前不得改动目标文件；恢复失败时保留回滚数据并返回非零退出码。

## 验收标准

- `data-store-copy` 中存在设计、恢复说明、清单、恢复脚本、拓扑 JSON 和引用底图。
- 备份 `traffic_map.json` 可解析，版本为 `3`，包含 13 条道路和 12 个摄像头。
- 清单中的每个文件都存在，文件大小和 SHA-256 与备份文件一致。
- 备份文件与捕获时的源文件逐字节一致。
- 恢复脚本可在隔离的临时项目目录完成校验、回滚快照和文件恢复，不修改真实项目数据。
- 恢复后的文件能够被当前版本 `TrafficMapModel` 加载，并保持 13 条道路、12 个摄像头及有效底图路径。

## 非目标

- 不备份车辆实时轨迹、道路热力状态、检测事件、视频或模型文件。
- 不修改拓扑格式、道路内容、摄像头绑定或应用代码。
- 不自动提交、推送或重启正在运行的服务。
