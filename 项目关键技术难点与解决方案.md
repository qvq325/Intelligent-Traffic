# 沙盘交通智控台：关键技术难点与解决方案总结

## 1. 项目背景与场景划分

本项目由早期 PyQt 桌面程序演进为浏览器前端与 FastAPI 后端分离的“沙盘交通智控台”。系统复用原有的车辆检测、中文车牌识别、白名单和道路拓扑能力，并增加了 Web 视频播放、REST API、地图编辑和运行态势展示。

结合当前源码、测试、12 路沙盘摄像头配置以及 `videos/` 中的演示素材，本文将项目能力归纳为以下 4 个完整场景：

| 场景 | 典型输入或素材 | 主要输出 |
| --- | --- | --- |
| 场景一：多源视频接入与实时监控 | 12 路 RTSP、上传的本地视频、路口/停车场/红绿灯等视频素材 | 浏览器实时画面、播放状态、截图、推理参数 |
| 场景二：车辆检测与持续追踪 | `车辆追踪.mp4`、路口和停车场视频 | 车辆类别、置信度、车辆框、稳定的局部跟踪 ID |
| 场景三：中文车牌识别与白名单联动 | `车牌识别.mp4`、`远距离车牌识别.mp4` | 车牌号码、颜色、置信度、车辆关联和白名单状态 |
| 场景四：道路轨迹与交通热力态势 | `热力图.mp4`、沙盘底图、摄像头和道路配置 | 全局车辆轨迹、道路车流量、占有率、热度和可视化地图 |

> 说明：`红绿灯.mp4` 当前属于场景一、二可接入的路口视频素材。现有代码能够对其中的车辆进行检测和跟踪，但尚未实现红灯、黄灯、绿灯状态分类，不能将其描述为已完成的信号灯识别场景。

## 2. 总体技术链路

```text
RTSP / 本地视频
        |
        v
VideoStreamService（采集线程、重连、暂停、帧同步）
        |
        +--> YOLO11m + ByteTrack --------> 车辆框、类别、局部 Track ID
        |                                      |
        +--> YOLO Pose + CRNN ------------> 车牌、颜色、置信度
                                               |
                                               +--> 白名单匹配
                                               |
                                               +--> TrafficMapModel
                                                      |
                                                      +--> 全局轨迹
                                                      +--> 流量/占有率/热度
        |
        +--> JPEG 编码 --> MJPEG --> 浏览器实时监控

FastAPI REST API <--> 前端状态管理 <--> Canvas 道路态势图
```

核心实现分布如下：

- 视频采集与发布：[backend/video_stream.py](backend/video_stream.py)
- Web 接口和文件上传：[backend/app.py](backend/app.py)
- 共享业务状态：[backend/state.py](backend/state.py)
- 车辆检测与跟踪：[vehicle_detector.py](vehicle_detector.py)
- 识别编排：[detection_processor.py](detection_processor.py)
- 中文车牌识别：[lpr_recognizer.py](lpr_recognizer.py)
- 白名单规则：[whitelist_manager.py](whitelist_manager.py)
- 道路、轨迹和热度模型：[traffic_map.py](traffic_map.py)
- 前端交互与地图绘制：[frontend/js/app.js](frontend/js/app.js)、[frontend/js/map-canvas.js](frontend/js/map-canvas.js)

## 3. 场景一：多源视频接入与浏览器实时监控

### 3.1 关键难点

1. **输入源异构**：系统既要读取 RTSP 网络流，也要播放帧率和编码各不相同的本地视频。
2. **网络流不稳定**：RTSP 可能出现连接超时、读帧失败、积帧和高延迟，不能因一次失败使整个服务退出。
3. **控制操作与采集并发**：用户可能随时切换视频源、暂停、继续或停止，采集线程必须避免资源泄漏和竞态条件。
4. **AI 推理与播放帧率冲突**：逐帧执行车辆检测和车牌识别会显著降低播放流畅度。
5. **浏览器不能直接可靠播放 RTSP**：需要把 OpenCV 读取的画面转换为浏览器可消费的协议。

### 3.2 对应解决方案

| 难点 | 解决方案 | 实现效果 |
| --- | --- | --- |
| RTSP 与本地文件统一接入 | 使用 OpenCV `VideoCapture` 和 FFmpeg 后端统一解码；通过路径判断区分本地文件与网络流 | 上层处理流程不依赖视频源类型 |
| 连接与读帧超时 | 配置 5 秒打开超时、3 秒读取超时；RTSP 失败后间隔 1 秒自动重连 | 避免线程长期卡死，短时断流可恢复 |
| 低延迟与硬件兼容 | 将缓冲区设置为 1；Windows 优先尝试 D3D11VA，失败后自动回退 CPU FFmpeg | 降低积帧延迟，同时保留无 GPU 环境兼容性 |
| 并发状态一致性 | 以后台工作线程负责采集；使用 `Condition + RLock` 保护共享状态，并用 `source_revision` 判断视频源是否已经变化 | 切流、暂停和停止不会与旧连接混用 |
| 推理吞吐不足 | 提供 `interval` 帧间隔，仅每 N 帧执行一次推理；中间帧复用上一次结果重新绘制 | 在识别实时性与播放流畅度之间可调节 |
| 模型启动耗时 | 仅在用户启用识别时懒加载模型；设备变化时才重建处理器 | 服务可以快速启动，不使用 AI 时不占用模型资源 |
| 浏览器播放 | 将标注帧编码为质量 86 的 JPEG，通过 `multipart/x-mixed-replace` 发布 MJPEG | 无浏览器插件即可显示实时画面 |
| 多客户端取帧 | 保存最新 JPEG 和递增帧序号，消费者通过条件变量等待新帧 | 避免每个浏览器连接重复采集和重复推理 |

本地视频还按源 FPS 节流，防止文件以处理器最大速度“瞬间播完”；播放结束与 RTSP 断流采用不同处理策略。FastAPI 同时提供选源、上传、暂停、停止、截图和状态接口，前端每秒轮询状态并更新画面及指标。

### 3.3 方案权衡

- MJPEG 实现简单、兼容性高、低延迟，但带宽和编码开销高于 H.264/WebRTC，适合当前单路选择式监控，不适合大规模多路并发墙。
- 当前配置包含 12 路 RTSP，但运行时一次选择并处理一路视频，不是 12 路同时推理。
- 上传文件采用分块写入、扩展名白名单、4 GiB 上限、安全文件名和随机前缀，避免一次性占满内存以及路径穿越。

## 4. 场景二：车辆检测与持续追踪

### 4.1 关键难点

1. **复杂交通画面中的车辆检出**：路口、停车场、桥面和远景画面存在尺度变化、遮挡和目标密集问题。
2. **检测框缺少时间连续性**：单帧 YOLO 只能给出目标框，无法判断相邻帧是否为同一辆车。
3. **切换摄像头后的 ID 污染**：复用旧跟踪器会把新画面中的车辆错误继承为旧 ID。
4. **设备差异**：项目需要在 CUDA GPU、Apple MPS 和纯 CPU 环境运行，并处理指定设备不存在的情况。
5. **推理结果需要供多个模块复用**：同一结果既要绘制到视频，又要进入车牌关联、地图轨迹和前端统计。

### 4.2 对应解决方案

| 难点 | 解决方案 | 实现效果 |
| --- | --- | --- |
| 车辆检测 | 使用 YOLO11m，只保留 car、motorcycle、bus、truck 等交通类别，并提供在线置信度阈值 | 减少无关目标和低质量框 |
| 连续跟踪 | 将 YOLO 检测框送入 Ultralytics ByteTrack，输出稳定的局部 `track_id` | 车辆在同一摄像头内可形成连续轨迹 |
| 多源隔离 | 以 `camera_id` 作为跟踪器键；切换视频源时清空跟踪状态 | 避免不同摄像头或不同视频文件之间串号 |
| 边界安全 | 对跟踪框执行图像边界裁剪，并过滤宽高无效的框 | 防止后续裁剪、绘制和坐标计算异常 |
| 设备适配 | 自动枚举 CUDA/MPS/CPU；指定 CUDA 编号无效时回退 `cuda:0`，GPU 不可用时回退 CPU | 同一代码可在开发机和部署机运行 |
| 统一数据契约 | 使用 `VehicleDetection` 和 `DetectionResult` 数据对象保存类别、框、置信度、ID、摄像头和时间戳 | 视频、车牌、地图、API 共用同一份结构化结果 |

`DetectionProcessor.process()` 将车辆检测、车牌识别、白名单匹配和标注绘制编排为一条管线。车辆框按白名单状态使用不同颜色，并在画面中显示跟踪 ID、类别和置信度。若车牌模型加载失败，车辆检测仍可继续运行，避免次要能力故障拖垮整个检测链路。

### 4.3 方案权衡

- ByteTrack ID 只保证单摄像头、单次连续播放期间的局部稳定性；跨摄像头身份由场景四的全局 ID 启发式逻辑处理。
- 跳帧推理时，中间帧复用旧检测框，能提高流畅度，但高速车辆的框可能短暂滞后。实际部署应按 GPU 性能和车速调整 `interval`。

## 5. 场景三：中文车牌识别与白名单联动

### 5.1 关键难点

1. **车牌目标小、倾斜且可能为双层牌**：直接对轴对齐矩形做 OCR 容易受透视和排版影响。
2. **中文车牌字符集与序列解码复杂**：需处理省份简称、字母数字、特殊车辆字符以及 CTC 重复字符和空白符。
3. **一帧内多车多牌关联**：必须知道每个识别到的车牌属于哪辆车。
4. **性能压力**：如果对每一个车辆 ROI 分别运行一次车牌模型，车辆数量增加时推理次数线性增长。
5. **业务规则不等于 OCR 结果**：识别文本还需要规范化、白名单匹配、持久化和在线维护。

### 5.2 对应解决方案

| 难点 | 解决方案 | 实现效果 |
| --- | --- | --- |
| 车牌定位和透视 | YOLO Pose 同时输出车牌框和四角关键点，再用透视变换将车牌矫正为矩形 | 提高斜拍车牌的 OCR 可读性 |
| 双层车牌 | 将上下区域裁剪、缩放并横向拼接为单行图像 | 复用同一 CRNN 序列识别模型 |
| 批量 OCR | 一次检测整帧中的全部车牌，统一缩放到 `168×48`，组成 batch 后送入 CRNN | 避免逐车重复运行车牌检测和 OCR |
| CTC 解码 | Softmax 后取每步最大概率，去除 blank 和连续重复字符，并计算有效字符平均置信度 | 得到可读车牌文本和可解释置信度 |
| 车牌与车辆关联 | 判断车牌框中心点是否落在车辆框内；同车存在多个候选时取置信度最高者 | 以较低计算成本完成多车多牌匹配 |
| 误识别控制 | 车辆检测阈值与车牌阈值独立配置，低于车牌阈值的结果直接丢弃 | 可按远景、近景分别调优 |
| 白名单匹配 | 依次执行精确匹配、最长前缀匹配和去除空格/连字符/点号后的归一化匹配 | 容忍部分输入格式差异 |
| 数据维护 | 白名单使用带锁管理器，支持增删、清空、启停、JSON 持久化和 REST API | 识别链路与业务配置实时联动 |

模型权重兼容也是生成过程中的实际难点。当前实现对上游权重中的旧层名前缀进行映射，以 `strict=False` 加载后再检查真正缺失的可训练参数和多余参数；只有兼容性检查通过才进入推理，避免“表面加载成功、实际层未生效”。CUDA 环境下还使用半精度批量推理降低显存和计算开销。

### 5.3 方案权衡

- “车牌中心落入车辆框”算法简单高效，但车辆严重重叠时可能关联错误；更高精度方案可结合车牌与车辆的几何位置、IoU、历史轨迹和车型先验进行加权匹配。
- 当前前缀规则会为白名单车牌建立长度不小于 3 的所有前缀，容错较强，也可能扩大误匹配范围。生产环境应按安全等级决定是否只允许显式通配前缀。
- 远距离车牌最终精度仍受原始像素、运动模糊和曝光影响，软件透视矫正不能恢复未采集到的字符细节。

## 6. 场景四：道路轨迹与交通热力态势

### 6.1 关键难点

1. **视频坐标与沙盘地图坐标不一致**：检测框位于二维图像中，而道路由归一化地图折线表示。
2. **摄像头只看到局部区域**：需要利用摄像头位置、朝向、视距和所关联道路推算车辆在地图上的位置。
3. **跨摄像头车辆身份连续**：ByteTrack 的局部 ID 在不同摄像头之间没有可比性。
4. **瞬时车辆数波动大**：直接用单次计数上色会导致热力图频繁闪烁。
5. **道路拓扑需要可编辑和持久化**：沙盘道路有双向车道、桥梁、停车场和服务区，不能只靠固定背景图。
6. **实时更新与配置编辑并发**：检测线程会更新轨迹，Web API 同时可能修改道路或摄像头配置。

### 6.2 对应解决方案

| 难点 | 解决方案 | 实现效果 |
| --- | --- | --- |
| 坐标映射 | 取车辆框底边纵坐标作为远近依据，将摄像头锚点投影到关联道路折线，再沿摄像头朝向移动 | 将视频检测近似映射到道路位置 |
| 弯曲道路 | 实现点到折线投影、折线总长度和按里程取点，不把道路简化成单一直线 | 轨迹可以沿沙盘曲线路网移动 |
| 方向判断 | 比较道路切向量与摄像头朝向向量的点积，确定沿道路正向或反向移动 | 支持顺/逆时针和双向道路 |
| 跨摄像头全局 ID | 优先以车牌复用全局 ID；无车牌时按不同摄像头、同车型、0.5~8 秒时间窗和 0.18 地图距离匹配最近轨迹 | 在没有专用 ReID 模型时保持基本连续性 |
| 轨迹生命周期 | 每个目标最多保存 40 个历史点；4 秒未更新则过期清理 | 控制内存并移除失效目标 |
| 流量统计 | 记录车辆首次进入道路的事件，只保留最近 60 秒 | 直接得到每分钟流量 |
| 占有率与热度 | `occupancy = min(1, 车辆数 / 道路容量)`；热度采用 `0.72 × 旧值 + 0.28 × 当前占有率` | 平滑瞬时抖动，同时保留变化响应 |
| 地图编辑 | 道路和摄像头坐标统一归一化；提供折线/曲线绘制、增改删、摄像头定位、底图上传和 JSON 持久化 | 适配不同沙盘底图与道路布局 |
| 并发保护 | `ApplicationState` 使用 `RLock` 保护地图快照、检测更新、配置修改和保存 | 避免迭代过程中集合变化或写入互相覆盖 |
| 前端绘制 | Canvas 分层绘制底图、道路热度、方向、摄像头视锥、历史轨迹和车辆标签 | 在一个视图中同时展示静态配置与运行态势 |

默认拓扑包含 18 条有方向的道路，摄像头配置与 12 路视频源名称对应。前端约每 1.8 秒拉取一次地图快照，只在“道路态势”页面激活时高频更新；浏览器切到后台后降低轮询频率，以减少无意义请求。

### 6.3 方案权衡

- 当前视频到地图的定位是基于框底边、摄像头朝向和视距的近似模型，并非相机标定后的单应性变换。若需要米级定位，应增加内外参标定和地面控制点。
- 跨摄像头全局 ID 使用车牌与时空邻近启发式，适合小型沙盘；真实道路部署应引入摄像头拓扑、行驶时间约束和车辆 ReID 特征。
- 热度是相对道路容量的平滑占有率，不等同于真实交通速度或拥堵指数。若要判定拥堵，还需融合平均速度、排队长度和信号周期。

## 7. 四个场景共用的工程难点

### 7.1 从桌面程序迁移到 Web 架构

迁移的核心不是简单替换界面，而是重新划分状态边界：

- FastAPI 管理视频、识别参数、白名单和地图 API。
- `ApplicationState` 统一持有视频服务、白名单和道路模型。
- 采集和推理保留在线程内，避免阻塞 HTTP 请求线程。
- 前端只消费 MJPEG 与 JSON，不直接依赖 OpenCV、PyTorch 或模型文件。
- 上传视频和地图放入 `runtime/`，道路与白名单分别持久化到 JSON。

这样既保留了原有算法模块，也让客户端从必须安装 Python/Qt 的桌面环境变为普通浏览器。

### 7.2 状态、错误和降级设计

- 视频连接状态、检测状态、分辨率、FPS 和识别结果由统一状态接口输出。
- LPR 不可用时仍保留车辆检测；硬件解码失败时回退软件解码；GPU 不可用时回退 CPU。
- 切流时释放旧 `VideoCapture`、清空旧帧和识别结果、重置局部跟踪器。
- 地图回调异常只更新检测状态，不使视频采集线程退出。
- API 对不存在的视频源、摄像头、道路、无截图、非法扩展名和超限文件返回明确 HTTP 状态码。

### 7.3 可验证性

测试按低成本、可重复的方式覆盖核心契约：

- CTC 去重与 blank 解码。
- 车牌中心点和车辆框的关联。
- 摄像头配置持久化、道路增改删和双向拓扑。
- 轨迹生成、道路流量/热度更新和超时清理。
- 视频暂停、切流、停止和 JPEG 发布。
- Web 首页、系统信息、识别设置、白名单、地图和校验错误 API。

测试刻意不连接真实 RTSP，也不加载大模型，以保证自动化测试快速稳定；这同时意味着部署前仍需补充真实摄像头、GPU、断流恢复、长时间运行和多浏览器并发测试。

## 8. 结论

项目最核心的技术工作，是把“视频读取、AI 识别、业务规则和地图态势”从各自独立的功能串成一条可实时运行、可配置、可降级、可在浏览器观察的完整链路。四个场景之间不是四套重复实现，而是逐层复用：

1. 场景一提供稳定、低延迟的视频和控制底座。
2. 场景二把视频帧转化为带局部身份的车辆目标。
3. 场景三为车辆补充车牌身份和白名单业务属性。
4. 场景四再把目标映射到道路空间，形成跨摄像头轨迹和交通热度。

当前方案适合沙盘演示和小规模交通监控验证。下一阶段若面向真实道路部署，优先级应是相机标定、跨镜 ReID、红绿灯状态识别、H.264/WebRTC 传输、全链路压力测试和运行指标监控。
